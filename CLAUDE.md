# lumi — Claude Code 핵심 지시서

## ⚡ 절대 규칙
1. 파일 수정 전 원본 반드시 읽기
2. str_replace/edit_block 최소 범위만 수정 (전체 재작성 금지)
3. 김현님 승인 없이 실제 파일 수정 금지
4. 추측·거짓말 금지 — 팩트만 보고
5. 작업 완료 후 반드시 배포
6. 코딩/파일수정 시작 전 구현 계획 먼저 보고 → 승인 후 진행
7. 작업 완료 시 변경 파일·배포 URL·다음 단계를 요약 보고

## 🪙 토큰 절약 규칙
- **기본 모델은 opusplan** — 계획 단계 자동 Opus, 실행 단계 자동 Sonnet 전환
- 작업 단위가 끝나면 반드시 `/clear` 로 컨텍스트 초기화
- 대화가 길어지면 `/compact` 로 압축 후 계속 진행
- 프롬프트 작성 시 프로젝트 전체가 아닌 **필요한 파일·함수만** 명시
- 에이전트에게 작업 지시 시 탐색 범위를 구체적으로 제한

## 🤖 에이전트 파이프라인
```
1단계: [아이디어 에이전트] → 개선 아이디어 제안
2단계: [검토 에이전트]    → 1단계 아이디어의 타당성·우선순위 검토
3단계: [김현님 1차 승인]  → 진행 여부 결정
4단계: [시범 에이전트]    → 프로토타입 제작 (파일 수정 금지)
5단계: [김현님 최종 승인] → 배포 여부 결정
6단계: [구현 에이전트]    → 실제 파일 반영 + 배포
7단계: [QA 에이전트]     → 버그·작동 검증
8단계: [최종 보고]       → 김현님께 변경사항·검증결과 보고
```

**에이전트별 역할 (혼동 금지)**
- 아이디어 에이전트: 개선 아이디어 제안 (근거 포함)
- 검토 에이전트: 아이디어 에이전트가 낸 아이디어를 검토 (코드 검토 아님)
- 시범 에이전트: 승인된 아이디어를 프로토타입으로 제작 (파일 수정 금지)
- 구현 에이전트: 최종 승인된 내용을 실제 파일에 반영 + 배포
- QA 에이전트: 배포 후 버그·작동 여부 검증 (코드 검토는 여기서)

**아이디어 에이전트 필수 규칙**
- docs/service.md의 완료 목록 반드시 읽고 중복 제안 금지
- docs/design.md의 디자인 스킬 적극 활용
- 뻔한 아이디어 금지. 김현님도 생각할 수 있는 수준은 가치가 없음
- 기존 틀에 얽매이지 말 것. 완전히 새로운 접근 환영
- 디자인 개선 + 기능 추가를 균형 있게 제안할 것
- 제약(메타 심사 중, 알림톡 검수 중 등)을 이유로 아이디어 범위를 좁히지 말 것
- 심사 완료 후 바로 실행할 수 있도록 미리 구상·준비하는 아이디어도 포함

**검토 에이전트 필수 규칙**
- 아이디어의 비즈니스 타당성 검토 (코드 검토 아님)
- 단순 상상이 아닌 실제 구현 가능 여부를 항상 검증
- "지금 제약이 있으니 보류"가 아니라 "지금 준비하고 심사 후 바로 실행" 관점으로 검토

**보고 규칙**
- 3단계(1차 승인), 5단계(최종 승인)만 김현님께 보고 후 대기
- 나머지 단계는 보고 없이 바로 진행
- 8단계(최종 보고)에서 변경사항·검증결과 종합 보고

**절대 규칙**
- 단계를 절대 건너뛰지 않는다
- 독단적으로 규칙을 깨지 않는다
- 각 에이전트는 자기 역할만 수행한다
