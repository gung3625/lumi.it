루미 작업 원칙. 짧게 유지. 작업하다 진짜 필요해지면 한 줄씩 추가.

LLM 코딩에서 흔한 실수를 줄이는 행동 원칙.
프로젝트별 추가 지침이 있으면 같이 본다.

> 📘 **프로젝트 인수인계 문서**: [HANDOFF.md](HANDOFF.md) — 아키텍처 / DB 스키마 / 환경변수 / 최근 작업 / 알려진 미완료 / 트러블슈팅 메모 모두 정리됨. 새 작업 시작 전 읽을 것.

**트레이드오프:** 이 원칙은 속도보다 신중함을 우선한다. 사소한 작업은 판단력 발휘.

## 1. 코딩 전 생각하기

추측하지 말 것. 헷갈리는 건 숨기지 말 것. 트레이드오프는 드러낸다.

작업 시작 전:
- 가정한 내용은 먼저 밝힌다. 불확실하면 묻는다.
- 해석이 여러 개면 몰래 고르지 말고 다 제시한다.
- 더 단순한 방법 있으면 말한다. 필요하면 반박한다.
- 모르는 게 있으면 멈춘다. 무엇이 헷갈리는지 짚고 묻는다.

## 2. 단순함이 최우선

문제 해결에 필요한 최소 코드. 추측성 코드 X.

- 요청한 기능 외에는 추가 X
- 일회용 코드에 추상화 X
- 요청하지 않은 "유연성"·"설정 가능성" X
- 불가능한 시나리오에 에러 처리 X
- 200줄 짠 게 50줄로 줄어든다면 다시 써라

스스로 묻기: "시니어 개발자가 이거 보면 과하다고 할까?" 그렇다면 단순화.

## 3. 외과 수술처럼 정확히 변경

필요한 것만 건드린다. 자기가 만든 어지름만 치운다.

기존 코드 수정 시:
- 인접 코드·주석·포맷팅을 "개선"하지 않는다
- 안 망가진 거 리팩토링하지 않는다
- 마음에 안 들어도 기존 스타일 따른다
- 무관한 dead code 발견 시 언급은 하되 삭제하지 않는다

내 변경으로 고아가 생기면:
- 내가 만든 변경이 안 쓰게 만든 import·변수·함수만 제거한다
- 원래 있던 dead code는 요청 없으면 안 건드린다

테스트: 모든 변경 라인이 사용자 요청에 직결돼야 한다.

## 4. 목표 주도 실행

성공 기준을 정의한다. 검증될 때까지 루프한다.

작업을 검증 가능한 목표로 변환:
- "검증 추가" → "잘못된 입력 테스트 작성 → 통과시키기"
- "버그 고치기" → "재현 테스트 작성 → 통과시키기"
- "X 리팩토링" → "전·후 테스트 통과 보장"

다단계 작업은 짧은 계획 명시:

```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

강한 성공 기준이면 독립적으로 루프 가능. 약한 기준("작동시켜줘")은 매번 명확화 요구.

---

**잘 작동하는 신호:** 디프에 불필요한 변경 ↓, 과복잡으로 인한 재작성 ↓, 명확화 질문이 실수 후가 아니라 구현 전에 옴.
